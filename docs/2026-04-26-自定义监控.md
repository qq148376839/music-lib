## 1. 概述

### 1.1 目标

让用户能监控任意歌单 URL，系统定时检查并自动下载新增歌曲——将 M3 的"榜单监控"能力从 3 平台 8 张固定榜单扩展到 7+ 平台的任意歌单。

### 1.2 范围

**覆盖**：
- 扩展现有 Monitor 模型支持歌单类型
- 歌单 URL 解析 + 定时拉取 + 去重下载
- 前端歌单监控创建/管理 UI

**不覆盖**：
- 歌手页面监控（无现成 provider 方法，需全新开发）
- 新增轮询间隔档位（6/12/24h 已满足需求）
- 歌单内容变更通知（平台无推送 API，只能轮询）

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 榜单监控 (chart) | M3 已有功能，监控平台官方排行榜 |
| 歌单监控 (playlist) | 本次新增，监控任意歌单 URL |
| 解析 (resolve) | 从歌单 URL 提取平台、歌单 ID、歌单名称等元数据 |

## 2. 假设审计

### 2.1 真实约束（不可改变）

| 约束 | 依据 |
|------|------|
| `GetPlaylistSongs(id)` 返回全部歌曲，无 limit 参数 | ProviderFuncs 接口定义（`provider.go:18`） |
| `ParsePlaylist(url)` 仅 7 个平台有实现 | netease/qq/kugou/kuwo/soda/fivesing/bilibili |
| 去重依赖 `(source, song_id)` 在 TaskRecord 表中有 status=done 记录 | M3 已验证的去重逻辑 |
| 调度器 1 分钟轮询，按 NextRunAt 判断 | M3 scheduler 架构 |

### 2.2 行业惯例（需要审视）

| 惯例 | 审视结果 | 是否采纳 | 理由 |
|------|----------|----------|------|
| 歌手监控作为独立功能 | 拒绝 | 否 | 没有平台提供 `GetArtistSongs` API。用户可以找到歌手的歌单/专辑 URL，用歌单监控间接实现。做歌手监控需要为每个平台写新的 provider 方法——投入产出比极低 |
| 歌单 + 榜单用不同数据表 | 拒绝 | 否 | 二者的调度、去重、入队逻辑完全相同，只是数据源的获取方式不同。拆两张表 = 重复代码 + 重复 API |
| 支持自定义 Cron 表达式 | 拒绝 | 否 | M3 已裁定 6/12/24h 三档够用，歌单监控沿用 |

### 2.3 关键决策

| 决策点 | 选择 | 备选方案 | 为什么选 A 不选 B |
|--------|------|----------|-------------------|
| 数据模型 | 扩展现有 Monitor 表加 `Type` + `SourceURL` 字段 | 新建 PlaylistMonitor 表 | 调度、去重、入队逻辑 100% 相同，加两个字段比维护两套 CRUD 成本低一个量级 |
| 歌曲获取方式 | playlist 类型用 `GetPlaylistSongs(id)` | 每次都调 `ParsePlaylist(url)` | `GetPlaylistSongs` 更轻量（只返回歌曲），`ParsePlaylist` 额外返回歌单元数据（创建时用一次就够了） |
| TopN 对 playlist 的含义 | 每次运行最多入队 TopN 首新歌 | 不限制 / 限制拉取数量 | `GetPlaylistSongs` 没有 limit 参数（会返回全部），限制入队数量是唯一可控的安全阀。默认 100，防止首次运行 1000 首歌淹没下载队列 |
| 歌单 URL 解析时机 | 创建时解析一次，存储 playlistID + 元数据 | 每次执行时重新解析 URL | 歌单 ID 不会变，反复解析 URL 是浪费。只在创建时解析 |

### 2.4 翻转条件

- **假设 1**：平台歌单 ID 长期不变。如果平台开始频繁重置歌单 ID，需要改为每次用 URL 重新解析。
- **假设 2**：`GetPlaylistSongs` 对所有歌单都有效。如果平台限制私密歌单访问，需要处理认证和错误。
- **假设 3**：用户歌单规模在合理范围内（< 2000 首）。如果出现万首级歌单，需要分页拉取。

## 3. 架构设计

### 3.1 整体架构

```
用户输入歌单 URL
       │
       ▼
[POST /api/monitors/resolve]  ← ParsePlaylist(url)
       │
       ▼
  返回: platform, playlistID, name, trackCount
       │
       ▼
[POST /api/monitors]  ← 创建 type=playlist 的 Monitor
       │
       ▼
[Scheduler tick()]
       │
       ├── type=chart    → GetChartSongs(chartID, topN)
       │
       └── type=playlist → GetPlaylistSongs(playlistID)
                                    │
                                    ▼
                            FilterDownloaded() 去重
                                    │
                                    ▼
                            取前 TopN 首新歌
                                    │
                                    ▼
                            dlMgr.EnqueueBatch()
```

### 3.2 模块划分

| 模块 | 变更 | 说明 |
|------|------|------|
| `internal/store/monitor.go` | 修改 | Monitor 模型加 Type + SourceURL 字段 |
| `internal/monitor/scheduler.go` | 修改 | ChartProvider 加 GetPlaylistSongs；execute() 分支处理两种类型 |
| `internal/api/handler_monitor.go` | 修改 | 新增 resolve 端点；CreateMonitor/UpdateMonitor 支持新字段 |
| `frontend/src/views/MonitorView.vue` | 修改 | 新增歌单 URL 输入 + 解析流程 |

### 3.3 核心流程

**创建歌单监控**：
1. 用户输入歌单 URL
2. 前端调 `POST /api/monitors/resolve` → 返回平台、ID、名称、歌曲数
3. 用户确认后调 `POST /api/monitors` 创建 type=playlist 的 Monitor

**定时执行（playlist 类型）**：
1. Scheduler `tick()` 找到到期的 playlist 类型 Monitor
2. 调 `provider.GetPlaylistSongs(playlistID)` 获取全部歌曲
3. 调 `FilterDownloaded(db, platform, songIDs)` 去重
4. 取前 TopN 首未下载歌曲
5. 调 `dlMgr.EnqueueBatch(newSongs, ...)` 入队
6. 记录 MonitorRun，更新 NextRunAt

## 4. API 规格

### 4.1 接口总览

| Method | Path | 描述 | 变更类型 |
|--------|------|------|----------|
| POST | /api/monitors/resolve | 解析歌单 URL，返回元数据 | **新增** |
| POST | /api/monitors | 创建监控规则 | **修改**（支持 type=playlist） |
| PUT | /api/monitors/:id | 修改监控规则 | **修改**（支持新字段） |
| GET | /api/monitors | 列出所有监控规则 | 不变（返回值含新字段） |
| DELETE | /api/monitors/:id | 删除监控规则 | 不变 |
| GET | /api/monitors/:id/runs | 执行历史 | 不变 |
| POST | /api/monitors/:id/trigger | 手动触发 | 不变 |
| GET | /api/charts?platform=X | 获取榜单列表 | 不变 |

### 4.2 接口详情

#### POST /api/monitors/resolve（新增）

- **描述**：解析歌单 URL，返回平台和歌单元数据，供创建监控前预览

**Request**:
```json
{
  "url": "https://music.163.com/#/playlist?id=3778678"
}
```

**Response (200)**:
```json
{
  "platform": "netease",
  "playlist_id": "3778678",
  "name": "热歌榜",
  "track_count": 200,
  "cover": "https://...",
  "creator": "网易云音乐"
}
```

**Error**:

| HTTP Status | 说明 | 客户端处理 |
|-------------|------|-----------|
| 400 | URL 为空或格式无效 | 提示用户检查 URL |
| 400 | 无法识别平台（URL 不属于已支持平台） | 提示支持的平台列表 |
| 400 | 该平台不支持歌单解析 | 提示用户换平台 |
| 502 | 平台 API 调用失败 | 提示稍后重试 |

#### POST /api/monitors（修改）

**Request**（新增字段用 **粗体** 标注）：
```json
{
  "name": "我的收藏",
  "platform": "netease",
  "chart_id": "3778678",
  "top_n": 100,
  "interval": 12,
  "**type**": "playlist",
  "**source_url**": "https://music.163.com/#/playlist?id=3778678"
}
```

字段规则：
- `type`：可选，值为 `"chart"`（默认）或 `"playlist"`
- `source_url`：当 type=playlist 时必填，存储原始歌单 URL
- `chart_id`：chart 类型存榜单 ID，playlist 类型存歌单 ID
- `top_n`：chart 类型默认 20（max 100），playlist 类型默认 100（max 500）
- `platform`：chart 类型限 netease/qq/kugou；playlist 类型支持 netease/qq/kugou/kuwo/soda/fivesing/bilibili

**新增校验规则**：
- type=playlist 时，source_url 不能为空
- type=playlist 时，top_n 上限放宽到 500
- 同 (platform, chart_id, type) 不允许重复创建

**Response**：同现有，返回完整 Monitor 对象（含新字段）

#### PUT /api/monitors/:id（修改）

支持更新 `top_n`、`interval`、`enabled`、`name`。
`type`、`platform`、`chart_id`、`source_url` 创建后不可修改（要改就删重建）。

## 5. 数据模型

### 5.1 表结构

#### monitors（修改）

新增字段用 **粗体** 标注：

| 字段 | 类型 | 约束 | 索引 | 说明 |
|------|------|------|------|------|
| id | uint | PK, autoincrement | | |
| name | string | NOT NULL | | 用户自定义名称 |
| platform | string | NOT NULL | idx_platform | 平台标识 |
| chart_id | string | NOT NULL | | chart: 榜单 ID / playlist: 歌单 ID |
| top_n | int | NOT NULL, default 20 | | 每次最多入队新歌数 |
| interval | int | NOT NULL, default 12 | | 轮询间隔（小时）：6/12/24 |
| enabled | bool | NOT NULL, default true | | 是否启用 |
| last_run_at | *time.Time | | | 上次执行时间 |
| next_run_at | time.Time | NOT NULL | | 下次执行时间 |
| **type** | **string** | **NOT NULL, default "chart"** | | **"chart" 或 "playlist"** |
| **source_url** | **string** | **默认空** | | **playlist 类型的原始 URL** |
| created_at | time.Time | | | |
| updated_at | time.Time | | | |

**新增唯一索引**：`UNIQUE(platform, chart_id, type)` — 防止重复创建

### 5.2 迁移策略

- `Type` 字段 default="chart"，现有数据自动兼容
- `SourceURL` 字段 default=""，现有数据不受影响
- GORM AutoMigrate 自动添加新列，无需手动迁移

## 6. 错误处理

### 6.1 resolve 端点错误

| 场景 | HTTP Status | 错误信息 | 处理方式 |
|------|-------------|----------|----------|
| URL 为空 | 400 | "url is required" | 前端校验 |
| 无法匹配平台 | 400 | "unsupported platform for URL" | 提示支持的平台 |
| ParsePlaylist 不可用 | 400 | "playlist parsing not supported for {platform}" | 提示用户 |
| 平台 API 失败 | 502 | "failed to resolve playlist: {error}" | 前端提示重试 |

### 6.2 scheduler 执行错误

| 场景 | 处理方式 | 日志 |
|------|----------|------|
| provider 无 GetPlaylistSongs | 跳过该 Monitor，MonitorRun 记录 failed | slog.Error |
| GetPlaylistSongs 返回空 | 正常完成，NewQueued=0 | slog.Info |
| GetPlaylistSongs 网络失败 | MonitorRun 记录 failed + error 信息 | slog.Error |

### 6.3 日志规范

```
slog.Info("monitor.execute", "monitor_id", m.ID, "type", m.Type, "platform", m.Platform)
slog.Info("monitor.playlist.fetched", "monitor_id", m.ID, "total", len(songs))
slog.Info("monitor.dedup", "monitor_id", m.ID, "new", newCount, "skipped", skippedCount)
slog.Error("monitor.execute.failed", "monitor_id", m.ID, "error", err)
```

## 7. 测试策略

### 7.1 单元测试

**Store 层**：
- `TestCreateMonitor_PlaylistType` — 创建 playlist 类型 Monitor，验证 Type 和 SourceURL 持久化
- `TestCreateMonitor_DuplicateCheck` — 同 (platform, chart_id, type) 重复创建返回错误
- `TestCreateMonitor_PlaylistTopN` — playlist 类型 TopN 默认 100，上限 500
- 现有 chart 测试全部保持不变（兼容性回归）

**Scheduler**：
- `TestExecute_PlaylistType` — mock GetPlaylistSongs，验证拉取→去重→入队流程
- `TestExecute_PlaylistType_EmptyPlaylist` — 空歌单正常完成
- `TestExecute_PlaylistType_AllDownloaded` — 全部已下载，NewQueued=0
- `TestExecute_PlaylistType_TopNLimit` — 新歌超过 TopN 时只入队 TopN 首

**API Handler**：
- `TestResolvePlaylist_Success` — 正常解析返回元数据
- `TestResolvePlaylist_InvalidURL` — 无效 URL 返回 400
- `TestResolvePlaylist_UnsupportedPlatform` — 不支持的平台返回 400
- `TestCreateMonitor_PlaylistFields` — 创建 playlist 类型验证字段

### 7.2 验收标准

1. 通过 resolve 接口解析一个网易云歌单 URL → 返回歌单名称和歌曲数
2. 创建一个 playlist 类型 Monitor（TopN=10, Interval=6）
3. 手动触发 → 最多 10 首新歌进入下载队列
4. 再次触发 → 已下载的歌曲全部去重跳过，只有新歌入队
5. 重启后监控规则和历史保留
6. 同一歌单 URL 不能重复创建监控
7. 前端可以输入歌单 URL → 预览 → 创建监控 → 查看执行历史

## 8. 实现计划

### 8.1 优先级排序

| 优先级 | 任务 | 依赖 | 说明 |
|--------|------|------|------|
| P0 | Monitor 模型加 Type + SourceURL 字段 | 无 | GORM AutoMigrate 自动处理 |
| P0 | ChartProvider 加 GetPlaylistSongs | 无 | scheduler.go 结构体扩展 |
| P0 | scheduler.execute() 分支处理 playlist 类型 | 上面两项 | 核心逻辑 |
| P0 | POST /api/monitors/resolve 端点 | 无 | 歌单 URL 解析 |
| P0 | CreateMonitor 支持 playlist 类型 + 唯一性校验 | 模型变更 | |
| P1 | 前端歌单 URL 输入 + resolve 预览 + 创建流程 | resolve API | |
| P1 | 单元测试 | 全部 P0 | |

### 8.2 里程碑对应

本设计属于 ROADMAP P1 项「自定义监控」，是 M3 榜单监控的自然延伸。

与现有模块的依赖：
- 依赖 M3 的 Monitor/MonitorRun/Scheduler 架构
- 依赖各平台的 `ParsePlaylist` + `GetPlaylistSongs` 实现
- 复用 M3 的去重逻辑 (`FilterDownloaded`)
- 复用现有下载管道 (`EnqueueBatch`)
