## 1. 概述

### 1.1 目标

修复三个已暴露的 bug：QQ 推荐歌单点击后报错、重启/重新部署后所有配置丢失、不支持推荐歌单的平台前端报错。

### 1.2 范围

| 覆盖 | 不覆盖 |
|------|--------|
| QQ 推荐歌单的 ID 兼容性修复 | 其他平台推荐歌单的新增支持 |
| 数据持久化的部署层修复 | SQLite 数据库迁移/备份机制 |
| 前端根据平台能力隐藏不支持的入口 | 前端 UI 重构 |
| 启动恢复日志完善 | 日志系统整体改造 |

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| `content_id` | QQ 音乐推荐歌单 API (`playlist.HotRecommendServer`) 返回的歌单 ID，`int64` 类型 |
| `disstid` | QQ 音乐歌单详情 API (`fcg_ucc_getcdinfo_byids_cp.fcg`) 要求的歌单 ID 参数 |
| `cdlist` | QQ 音乐歌单详情 API 响应中的歌曲列表数组 |
| capabilities | 后端 `/providers` 接口返回的平台能力声明 |

## 2. 假设审计

### 2.1 真实约束（不可改变）

| 约束 | 依据 |
|------|------|
| QQ 音乐推荐 API 返回 `content_id`，不返回 `disstid` | 代码验证：`qq/qq.go:288` |
| 旧版歌单详情接口 `fcg_ucc_getcdinfo_byids_cp.fcg` 用 `disstid` 查询 | 代码验证：`qq/qq.go:349` |
| 部分 `content_id` 不是有效的 `disstid` | 用户反馈 + 错误日志 |
| Docker bind mount `./data:/data` 不会被 `docker compose down` 删除 | Docker 文档：bind mount 不受 `-v` flag 影响 |
| NAS 部署脚本执行 `git reset --hard FETCH_HEAD` | 记忆记录：`feedback_docker_build.md` |
| `/providers` 接口已返回 `playlist_recommended` 能力字段 | 代码验证：`internal/api/handler_search.go:41` |

### 2.2 行业惯例（需要审视）

| 惯例 | 审视结果 | 是否采纳 | 理由 |
|------|----------|----------|------|
| 用统一的 `musicu.fcg` POST 接口替代所有旧 GET 接口 | 新接口更统一但模块名无公开文档 | 不采纳 | 旧接口对搜索歌单仍然可用，只对推荐歌单的 `content_id` 失败；修改范围最小化 |
| 前端做 feature flag 系统 | 过度设计 | 不采纳 | 后端已有 `/providers` 能力声明，前端读取即可 |
| 用 Docker named volume 替代 bind mount | 需要修改部署流程 | 不采纳 | bind mount 对 NAS 场景更直观（用户可直接访问数据目录） |

### 2.3 关键决策

| 决策点 | 选择 | 备选方案 | 为什么选 A 不选 B |
|--------|------|----------|-------------------|
| Bug 1 修复策略 | `fetchPlaylistDetail` 失败时 fallback 到 `musicu.fcg` POST 接口 | 全部迁移到新接口 | 搜索歌单用旧接口没问题，只有推荐歌单的 `content_id` 有问题；最小修改原则 |
| Bug 2 `.gitignore` 范围 | 忽略整个 `data/` 目录 | 只忽略 `data/*.db` 和 `data/*_cookie.json` | `data/` 是纯运行时数据目录，不应有任何文件进入 git |
| Bug 2 `docker-compose.yml` | 默认启用 `build: .` | 保持注释 | NAS 场景本地构建是主要使用方式；部署脚本依赖 `docker compose build` |
| Bug 3 前端能力适配 | 前端从已有 `/providers` 接口读取能力 | 新增专用 `/api/capabilities` 接口 | 接口已存在且字段已覆盖，无需新增 |

### 2.4 翻转条件

- 假设 1：QQ 音乐 `musicu.fcg` POST 接口长期可用。如果 QQ 音乐全面启用签名验证（`sign` 参数），所有非官方调用都会失败，此时需要重新评估整个 QQ 平台的可用性。
- 假设 2：`content_id` 可通过 `musicu.fcg` 的 `music.srfDissInfo.DissInfo` 模块查询歌单详情。如果该模块不接受 `content_id`，需要在推荐列表阶段就获取完整歌曲信息。

## 3. 架构设计

### 3.1 修改范围

本轮不涉及新模块，仅修改现有模块：

```
qq/qq.go                           ← Bug 1：新增 fallback 函数
.gitignore                         ← Bug 2：忽略 data/ 目录
docker-compose.yml                 ← Bug 2：启用 build
cmd/server/main.go                 ← Bug 2：启动恢复日志
internal/api/handler_search.go     ← Bug 3：扩展 /providers 响应
frontend/src/views/SearchView.vue  ← Bug 3：根据能力隐藏入口
```

### 3.2 Bug 1：QQ 推荐歌单修复 — 核心流程

当前流程（出错路径）：
```
前端点击推荐歌单
→ GET /api/playlist/songs?source=qq&id=<content_id>
→ GetPlaylistSongs(content_id)
→ fetchPlaylistDetail(content_id)
→ fcg_ucc_getcdinfo_byids_cp.fcg?disstid=<content_id>
→ 响应 cdlist 为空
→ 500 "playlist not found (empty cdlist)"
```

修复后流程：
```
前端点击推荐歌单
→ GET /api/playlist/songs?source=qq&id=<content_id>
→ GetPlaylistSongs(content_id)
→ fetchPlaylistDetail(content_id)
→ fcg_ucc_getcdinfo_byids_cp.fcg?disstid=<content_id>
→ 如果 cdlist 为空：
    → fallback: fetchPlaylistDetailV2(content_id)
    → musicu.fcg POST {module: "music.srfDissInfo.DissInfo", method: "CgiGetDiss"}
    → 返回歌曲列表
→ 200 OK
```

### 3.3 Bug 1：详细设计

**新增函数** `fetchPlaylistDetailV2(id string)` in `qq/qq.go`：

请求：
```
POST https://u.y.qq.com/cgi-bin/musicu.fcg
Content-Type: application/json
```

请求体：
```json
{
  "comm": {
    "ct": 24,
    "cv": 10000
  },
  "req": {
    "module": "music.srfDissInfo.DissInfo",
    "method": "CgiGetDiss",
    "param": {
      "disstid": <id_as_int>,
      "onlysonglist": 0,
      "song_begin": 0,
      "song_num": 100
    }
  }
}
```

> **实现注意**：模块名 `music.srfDissInfo.DissInfo` 来自社区逆向分析，无官方文档。实现时需先用 `curl` 验证该模块是否可用。如果不可用，尝试以下备选模块名：
> - `srf_diss_info.DissInfoServer`
> - `playlist.SongListService`
> 如果所有模块都不可用，采用 Plan B：在 `GetRecommendedPlaylists` 阶段，请求歌单时顺带拉取每个歌单的前 N 首歌曲（通过修改 `get_hot_recommend` 的 param）。

响应解析字段映射（预期结构，需验证）：
```
resp.req.data.dirinfo.title    → Playlist.Name
resp.req.data.dirinfo.picurl   → Playlist.Cover
resp.req.data.dirinfo.desc     → Playlist.Description
resp.req.data.songlist[].mid   → Song.ID (songmid)
resp.req.data.songlist[].name  → Song.Name
resp.req.data.songlist[].singer[].name → Song.Artist
resp.req.data.songlist[].album.name    → Song.Album
resp.req.data.songlist[].interval      → Song.Duration
```

**修改 `fetchPlaylistDetail`**：在 `len(resp.Cdlist) == 0` 处不直接返回错误，改为调用 `fetchPlaylistDetailV2`。

修改位置：`qq/qq.go` 约第 410 行

修改前：
```go
if len(resp.Cdlist) == 0 {
    return nil, nil, errors.New("playlist not found (empty cdlist)")
}
```

修改后：
```go
if len(resp.Cdlist) == 0 {
    return q.fetchPlaylistDetailV2(id)
}
```

### 3.4 Bug 2：数据持久化修复 — 详细设计

三个修改点：

**修改 1：`.gitignore` 增加 `data/` 目录**

```diff
+ # 运行时数据（SQLite + cookie + 下载缓存）
+ data/
- # 下载缓存
- data/downloads/
- data/video_output/
```

**修改 2：`docker-compose.yml` 启用 `build: .`**

```diff
  services:
    music-lib:
-     image: rio22/music-lib:latest
-     # build: .  # uncomment for local build
+     build: .
+     image: rio22/music-lib:latest
```

这样 `docker compose build` 会从本地 Dockerfile 构建，`docker compose pull` 可拉取远程镜像。NAS 部署脚本的 `docker compose build --no-cache` 将正常工作。

**修改 3：启动恢复摘要日志**

在 `cmd/server/main.go` 的启动流程中，任务和监控恢复后增加摘要日志。

新增位置：`main.go` 约第 128 行之后

```
slog.Info("data restored",
    "tasks", len(existingTasks),
    "netease_cookie", loggedIn_netease,
    "qq_cookie", loggedIn_qq,
)
```

监控恢复后（第 153 行之后）：
```
slog.Info("monitors loaded", "count", monitorCount)
```

这让运维可以通过 `docker logs` 一眼看出数据是否恢复成功。

### 3.5 Bug 3：平台能力适配 — 详细设计

**后端**：扩展 `/providers` 响应，补充排行榜相关能力字段。

修改位置：`internal/api/handler_search.go` 的 `providerInfo` struct

```diff
  type providerInfo struct {
      Name                string `json:"name"`
      Search              bool   `json:"search"`
      Download            bool   `json:"download"`
      Lyrics              bool   `json:"lyrics"`
      Parse               bool   `json:"parse"`
      PlaylistSearch      bool   `json:"playlist_search"`
      PlaylistSongs       bool   `json:"playlist_songs"`
      PlaylistParse       bool   `json:"playlist_parse"`
      PlaylistRecommended bool   `json:"playlist_recommended"`

  }
```

赋值处新增：
```go
Charts:     p.GetCharts != nil,
ChartSongs: p.GetChartSongs != nil,
```

**前端**：`SearchView.vue` 中根据能力隐藏推荐入口。

修改位置：`frontend/src/views/SearchView.vue`

修改逻辑：
1. 从 store 或 API 获取当前平台的 `playlist_recommended` 能力
2. "推荐" 按钮的 `v-if` 绑定到该能力值
3. 当 `source` 为 `"all"` 时，推荐按钮始终显示（内部只请求支持的平台）

前端需要的数据流：
```
组件挂载 → 调用 GET /providers → 缓存到 store.providers
切换平台 → 从 store.providers 查找当前平台的 playlist_recommended
```

## 4. API 规格

### 4.1 接口变更总览

| Method | Path | 变更类型 | 描述 |
|--------|------|----------|------|
| GET | /providers | 修改 | 新增 `charts`、`chart_songs` 字段 |
| GET | /api/playlist/songs | 行为修改 | QQ 推荐歌单不再返回 500 |

### 4.2 GET /providers — 变更后响应

**Response (200)**:

### 4.3 GET /api/playlist/songs — 行为变更

变更前：QQ 推荐歌单 → 500 "playlist not found (empty cdlist)"

如果 fallback 也失败：

**Error**:
| HTTP Status | 错误信息 | 触发条件 | 客户端处理 |
|-------------|---------|----------|-----------|
| 500 | "playlist not found (empty cdlist, fallback also failed)" | 旧接口和新接口都查不到 | 显示错误提示 |

## 6. 错误处理

### 6.1 Bug 1 错误处理策略

| 场景 | 旧行为 | 新行为 |
|------|--------|--------|
| `content_id` 在旧接口返回空 cdlist | 直接返回 500 | 尝试 fallback V2 接口 |
| fallback V2 也失败 | N/A | 返回 500 + 明确错误信息 |
| fallback V2 响应格式异常 | N/A | JSON 解析错误 → 返回 500 |

### 6.2 日志规范

新增的日志点：

| 级别 | 位置 | 消息 | 字段 |
|------|------|------|------|
| INFO | 启动恢复 | "data restored" | tasks, netease_cookie, qq_cookie |
| INFO | 监控加载 | "monitors loaded" | count |
| WARN | fallback 触发 | "qq playlist cdlist empty, trying v2" | id |
| ERROR | fallback 失败 | "qq playlist v2 also failed" | id, error |
