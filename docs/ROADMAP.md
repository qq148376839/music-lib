# music-lib ROADMAP

> 版本：1.1
> 日期：2026-04-26
> 更新：新增 M6 稳定性修复（推荐歌单 + 持久化 + 平台能力适配）
> 方法论：马斯克五步法（先删、再简化、再优化、最后才自动化）

---

## 核心问题陈述

**music-lib 的存在是为了让 NAS 用户能够自动构建并维护一个元数据完整的本地音乐库，而不需要手动搜索下载、手动补标签、手动监控榜单。**

---

## 现状盘点（2026-04-24）

已经做完的（别碰）：
- ✅ 11 个平台的搜索 + 下载（search、GetDownloadURL）
- ✅ 歌单解析 + 批量下载（ParsePlaylist、GetPlaylistSongs）
- ⚠️ QQ 音乐 + 网易云 QR 码登录（Cookie 持久化到磁盘，但依赖 Python+Playwright，Docker 内不可用 → M5 修复）
- ✅ NAS 下载队列（goroutine 池 + 并发控制）
- ✅ 基础 Web UI（vanilla HTML/JS，能搜能下）
- ✅ 音质全局设置（lossless/high/standard）

完全缺失的：
- ❌ SQLite 持久化（任务历史一重启就丢）
- ❌ Gin 路由框架（当前是 net/http 平铺）
- ❌ 音乐刮削（下载的文件无 ID3 标签、无封面、无歌词）
- ❌ 榜单监控 + 定时下载
- ❌ Docker 多阶段构建 + docker-compose
- ❌ 结构化日志（当前是 log.Printf）

## 功能优先级裁定

### P0 — 没有它系统解决不了核心问题

| 功能 | 为什么是 P0 | 删掉的代价 |
|------|------------|-----------|
| SQLite 任务持久化 | 下载任务重启丢失 = 用户不敢重启系统 | 系统不可靠 |
| Gin 框架迁移 | Monitor CRUD 等 RESTful 路由在 net/http 下会变成屎山 | 后续功能开发成本翻倍 |
| 音乐刮削（ID3 + 封面 + 歌词） | 下载的文件在任何播放器里都是 "未知 - 未知" | 文件库没有可用性 |
| Docker 多阶段构建 + Compose | NAS 部署的唯一方式 | 系统无法在 NAS 上跑 |
| 榜单监控（QQ + 网易，共 6-8 张榜） | 自动化是 NAS 系统的核心价值 | 退化为手动下载器 |

### P1 — 有价值，P0 全部完成后再做

| 功能 | 为什么是 P1 | 保留理由 |
|------|------------|---------|
| 结构化日志（slog） | 不影响功能，影响调试效率 | Go 1.21 stdlib，2 小时改完 |
| Web UI 升级（Vue 3） | 现有 vanilla UI 够用，只是不好看 | 功能完整后再打磨体验 |
| 自定义监控（关注特定歌手/歌单） | 需要 scheduler 稳定后才能做 | M3 完成后自然延伸 |

### 删除 — 说不清存在理由

| 功能 | 删除原因 |
|------|---------|
| **Acoustid 音频指纹识别** | 需要 chromaprint 外部库 + 调用 AcoustID API（有配额限制）。80% 场景用歌名搜索比指纹更快更准。没有它，刮削功能照样完整。 |
| **所有 11 个平台的榜单** | 大部分平台无公开榜单 API（Bilibili/Jamendo/JOOX/5sing 等）。维护 11 个平台的榜单爬虫 = 无止境的反爬对抗。只做 QQ + 网易 + 酷狗（3 个，已覆盖国内 80% 用户），其他平台列为 P1 下一轮再议。 |
| **Emby/Jellyfin 直接集成** | 用户需求文档未提及。Emby/Jellyfin 自己扫目录 + 刮削，music-lib 只需把文件放对位置 + 标签写好，二者自然配合。不需要 API 对接。 |
| **多源智能打分刮削** | music-tag-web 风格的"搜 5 个平台打分取最优"复杂度极高。用下载来源平台的元数据 = 最简方案，覆盖 95% 场景。打分逻辑列为 P1。 |

## Milestone 总览

| # | 名称 | 核心交付 | 验收标准 |
|---|------|---------|---------|
| M0 | 音质修复 | 实际音质显示 + 音质升级覆盖 + favicon | 配置无损但降级时有提示；登录VIP后重下可升级已有文件 |
| M1 | 基建重构 | Gin + SQLite + Docker | `docker-compose up` 可运行；重启后任务历史不丢 |
| M2 | 音乐刮削 | ID3 + 封面 + 歌词 | 下载任何歌曲 → 在播放器里看到正确标签和封面 |
| M3 | 榜单监控 | 定时下载 + 去重 | 配置一张榜 → 系统定时下载 Top 20，不重复 |
| M4 | UI 升级 | Vue 3 SPA + 监控管理页 | 所有功能可通过浏览器操作，无需直接调 API |
| M5 | Go 原生登录 | 去 Python 化，纯 Go QR 登录 | Docker 容器内扫码登录正常工作，无 Python 依赖 |
| M6 | 稳定性修复 | 推荐歌单修复 + 持久化修复 + 平台能力适配 | QQ 推荐歌单可点击；NAS 重启后配置不丢；不支持的功能入口隐藏 |

## 不做什么（明确排除）

- Acoustid 音频指纹：下一轮再议，先证明基础刮削的价值
- 11 个平台的榜单：Bilibili/5sing/Jamendo 无公开榜单 API，无法实现
- Emby/Jellyfin API 集成：文件放对 + 标签写对就够了
- 多源打分刮削：过度设计，先上单源
- Windows 桌面客户端：NAS 场景不需要

## 约束与技术决策（已锁定）

| 决策 | 内容 | 为什么不改 |
|------|------|----------|
| 后端语言 | Go，复用现有 11 个 provider 代码 | 2 月技术评估已定，加密模块迁移风险极高 |
| Web 框架 | Gin | 路由中间件 + binding 是 RESTful 的必需品 |
| 存储 | SQLite + GORM | 单文件，NAS 友好，无需外部数据库 |
| 部署 | Docker 多阶段构建，单容器 | NAS 场景唯一可接受方案 |
| 前端（最终态） | Vue 3 + Vite + Naive UI | M4 再做，M1-M3 沿用现有 vanilla UI |
