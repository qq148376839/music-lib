## 1. 概述

### 1.1 目标

让用户知道实际下载到了什么音质，并能在条件改善后（如登录 VIP）将已有低音质文件升级为高音质。

### 1.2 范围

| 覆盖 | 不覆盖 |
|------|--------|
| Provider 回写实际音质 | SQLite 持久化音质记录（M1 范围） |
| 前端显示实际音质 + 降级提示 | 自动检测"哪些歌可升级"（需调 API，太重） |
| 文件级音质比较 + 安全替换 | 跨目录文件去重 |
| 历史页面升级按钮 + 全部升级 | 定时自动升级（过度自动化） |
| Favicon | PWA manifest（后续再说） |

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 请求音质 | 用户在前端选择的音质档位：lossless / high / standard |
| 实际音质 | Provider 降级后实际获得的格式和码率 |
| 音质分数 | 用于比较新旧文件质量的数值，分数越高 = 音质越好 |
| 安全替换 | 先写临时文件 → 成功后原子替换，中途失败不丢旧文件 |

## 2. 假设审计

### 2.1 真实约束（不可改变）

| 约束 | 依据 |
|------|------|
| 未登录时平台只提供低音质 | QQ/网易的 API 行为，已验证 |
| 文件后缀决定播放器识别 | NAS 播放器（Plex/Navidrome）按后缀判断格式 |
| Go 标准库无音频元数据读取能力 | 需第三方库才能读 ID3/Vorbis 标签 |
| NAS 文件系统支持原子 rename | ext4/btrfs 上 `os.Rename` 是原子操作（同分区内） |

### 2.2 行业惯例（需要审视）

| 惯例 | 审视结果 | 是否采纳 | 理由 |
|------|----------|----------|------|
| 读文件 ID3 标签获取精确码率 | 拒绝 | 否 | 需引入 dhowden/tag 或类似库，M0 不加新依赖。用 Provider 回写的值 + 文件名后缀启发式，覆盖 95% 场景 |
| 保留旧文件备份（.bak）| 拒绝 | 否 | NAS 空间有限，替换即删旧。安全性通过临时文件机制保证 |
| 升级前先调 API 检查可用音质 | 拒绝 | 否 | N 首歌 = N 次 API 调用，太重。直接重新下载，让比较逻辑判断 |

### 2.3 关键决策

| 决策点 | 选择 | 备选方案 | 为什么选 A 不选 B |
|--------|------|----------|-------------------|
| 音质比较方式 | 格式优先级 + 文件大小启发式 | 读文件 ID3 元数据 | 不加新依赖，Provider 回写后新下载有精确值，旧文件用启发式兜底 |
| 已有文件查找方式 | `filepath.Glob` 匹配 `{Artist} - {Name}.*` | 只匹配精确 destPath | 后缀可能不同（.mp3 vs .flac），必须通配匹配 |
| WriteSongToDisk 返回值 | 结构体 `WriteResult` | 增加更多 bool 参数 | bool 参数超过 2 个就是灾难，结构体可扩展 |
| 升级触发方式 | 用户手动点按钮，重新入队 | 自动检测 + 静默升级 | 自动化前提是逻辑已验证正确，M0 先做手动 |
| 升级 API 粒度 | 批量接口，接受 task ID 列表 | 每首歌单独调接口 | 500 首歌 = 500 次请求，不现实 |

### 2.4 翻转条件

- 假设 1：Provider 能在 `GetDownloadURL` 内确定实际音质。如果某个平台的 API 不返回质量信息，该平台无法显示实际音质。
- 假设 2：`os.Rename` 在 NAS 文件系统上是原子的。如果用户的 NAS 使用跨设备 mount（如 NFS 到另一台机器的磁盘），rename 可能非原子。
- 假设 3：M1（SQLite）完成后，音质信息可持久化，升级比较从启发式变为精确。本设计需要兼容这个演进路径。

### 3.1 数据流

```

│        │                      │          │                     │ (QQ/网易) │
│        │                      │          │  ← URL + 回写       │          │
│        │                      │          │  Song.Bitrate/Ext   │          │
│        │                      │          │                     └──────────┘
│        │                      │          │
│        │                      │   Enqueue(song)                ┌──────────┐
│        │                      │ ─────────────────────────────→ │ Manager  │
│        │                      └──────────┘                     │          │
│        │                                                       │ runTask  │
│        │                                                       │   │      │
│        │                                                       │   ▼      │
│        │                                        ┌──────────────┤ Writer   │
│        │                                        │              │          │
│        │                                        │  查找已有文件  │          │
```

### 3.2 模块划分

| 模块 | 职责 | 改动类型 | 包路径 |
|------|------|----------|--------|
| Provider (QQ/网易/酷狗) | GetDownloadURL 回写实际 Bitrate/Ext | 修改 | `qq/`, `netease/`, `kugou/` |
| model.Song | 新增 QualityString() 辅助方法 | 修改 | `model/` |
| download.Writer | 文件查找 + 音质比较 + 安全替换 | 重构 | `download/` |
| download.Task | Task 增加音质字段；新增 WriteResult 类型 | 修改 | `download/` |
| download.Manager | 新增 EnqueueUpgrade 方法 | 修改 | `download/` |
| API Handler | 新增 upgrade 接口 | 修改 | `internal/api/` |
| 前端 | 音质显示 + 升级按钮 | 修改 | `web/` |

### 3.3 核心流程

#### 流程 A：正常下载（含音质回写）

1. 前端发起 `POST /api/nas/download?quality=lossless`
2. Handler 将 quality 写入 `song.Extra["quality"]`，将 quality 值记录到 Task.RequestedQuality
3. Manager.Enqueue → runTask → Provider.GetDownloadURL
4. **Provider 逐级尝试音质，命中后回写 `song.Bitrate` 和 `song.Ext`**（新增逻辑）
5. Writer.WriteSongToDisk：
   - Glob 查找 `{dir}/{Artist} - {Name}.*`
   - 无匹配 → 正常下载 → 返回 `ActionNew`
   - 有匹配 → 比较音质分数 → 高于旧文件 → 安全替换 → 返回 `ActionUpgraded`
   - 有匹配 → 音质分数不高于旧文件 → 返回 `ActionSkipped`
6. Task 记录 ActualQuality 字符串
7. 前端轮询拿到 Task，显示实际音质 + 状态标识

#### 流程 B：历史升级

1. 用户点击"升级"或"全部升级"
2. 前端发起 `POST /api/nas/download/upgrade`，body 带 task ID 列表 + quality
3. Handler 调用 Manager.EnqueueUpgrade
4. Manager 遍历 task IDs，查找已完成任务，取其 Song + Source，设置新 quality
5. 用对应 Provider 的 getURL/getLyrics 重新 Enqueue
6. 后续走流程 A 的步骤 3-7

- **Method**: POST
- **Path**: /api/nas/download/upgrade
- **描述**: 将已完成的下载任务重新入队，以当前音质设置尝试获取更高质量版本

**Query Parameters**:

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| quality | string | 否 | 目标音质（lossless/high/standard），默认 lossless |

**Request Body**:

```json
{
  "task_ids": ["t-abc123", "t-def456"]
}
```

- `task_ids` 为空数组或省略 → 升级所有已完成（status=done, skipped=false）的任务

**Response (200)**:

```json
{
  "batch_id": "b-xyz789",
  "queued": 3,
  "skipped": 2,
  "errors": [
    {"task_id": "t-xxx", "reason": "task not found"},
    {"task_id": "t-yyy", "reason": "task still running"}

}
```

| 字段 | 说明 |
|------|------|
| batch_id | 升级批次 ID，用于跟踪进度 |
| queued | 成功入队的数量 |
| skipped | 跳过的数量（任务不存在、仍在运行、无 source 信息等） |
| errors | 每个跳过任务的具体原因 |

**Error**:

| HTTP Status | 说明 | 客户端处理 |
|-------------|------|-----------|
| 400 | 请求体格式错误 | 显示错误提示 |
| 404 | 没有找到任何可升级的任务 | 显示"没有可升级的任务" |

## 6. 模块详细设计

### 6.1 Provider 回写逻辑

#### QQ 音乐 (qq/qq.go)

在 `GetDownloadURL` 的降级循环中，命中某个 rate 后、return 前，回写：

| Rate | Song.Ext | Song.Bitrate |
|------|----------|-------------|
| F000 | "flac" | 0（无损不用码率） |
| M800 | "mp3" | 320 |
| M500 | "mp3" | 128 |
| C400 | "m4a" | 96 |

改动位置：`for _, rate := range rates` 循环内，`return url, nil` 之前。

#### 网易云 (netease/netease.go)

在 `GetDownloadURL` 的降级循环中，命中某个 br 后、return 前，回写：

| br | Song.Ext | Song.Bitrate |
|----|----------|-------------|
| 999000 | "flac" | 0 |
| 320000 | "mp3" | 320 |
| 128000 | "mp3" | 128 |

改动位置：`for _, br := range brList` 循环内，`return resp.Data[0].URL, nil` 之前。

注意：网易 API 响应中有 `resp.Data[0].Br` 字段，可以用 API 返回的真实值替代硬编码的 br 映射。优先使用 API 返回值。

#### 酷狗 (kugou/kugou.go)

`fetchSongInfo` 返回的结构体中应有 bitrate 信息。如果有，回写到 Song。如果没有，Ext 从 URL 后缀推断，Bitrate 置 0。

### 6.2 Writer 重构：WriteSongToDisk

**当前签名**：
```go
func WriteSongToDisk(baseDir string, song *model.Song, audioURL, lyrics string, progressFn func(int64)) (string, bool, error)
```

**新签名**：
```go
func WriteSongToDisk(baseDir string, song *model.Song, audioURL, lyrics string, progressFn func(int64)) (WriteResult, error)
```

**核心逻辑**：

```
1. 构建目标目录 dir = buildSongDir(baseDir, song)
2. 构建目标文件名 newFilename = song.Filename()  // "Artist - Name.flac"
3. 构建基名模式 basePattern = SanitizeFilename(Artist + " - " + Name) + ".*"
4. Glob 搜索 dir 下匹配 basePattern 的文件

5. if 无匹配文件:
     → 正常下载到 dir/newFilename
     → 返回 WriteResult{Action: ActionNew}

6. if 有匹配文件:
     → 取音质分数最高的作为 existingFile
     → 计算 existingScore = qualityScore(existingExt, 0, existingSize)
     → 计算 newScore = qualityScore(song.Ext, song.Bitrate, 0)
     
     6a. if newScore <= existingScore:
           → 返回 WriteResult{Action: ActionSkipped}
     
     6b. if newScore > existingScore:
           → 下载到 dir/newFilename.tmp（临时文件）
           → 下载成功 → os.Rename(tmpPath, dir/newFilename)
           → 如果 existingFile 和 newFilename 路径不同 → os.Remove(existingFile)
           → 返回 WriteResult{Action: ActionUpgraded, PreviousExt: ..., PreviousSize: ...}
```

**边界条件处理**：

| 场景 | 处理 |
|------|------|
| 同后缀同名文件（精确匹配） | 比较文件大小。新文件明显更大（>20%）→ 替换；否则跳过 |
| 多个匹配文件（.mp3 和 .flac 都存在） | 取分数最高的比较。如果最高的不低于新文件 → 跳过 |
| 下载临时文件时失败 | 删除 .tmp 文件，返回错误，旧文件不受影响 |
| Rename 后 Remove 旧文件失败 | 新文件已就位，旧文件残留。日志 warn，不报错。下次升级时旧文件不影响（新文件分数更高） |
| song.Bitrate 为 0 且 ext 相同 | 退化为文件大小比较（新文件 size 用 Content-Length 响应头或下载后实际大小） |

**关于新文件大小的获取**：

新文件的 size 在下载前未知（Provider 只返回 URL）。处理方式：
- 先用 `qualityScore(ext, bitrate, 0)` 比较格式和码率
- 如果格式和码率都相同（score 相等），说明可能是同一音质来源，**跳过**
- 不做同格式同码率的文件大小比较（意义不大，且需要先下载才知道大小）

这比 milestone 文档里的"同格式比大小"更简单。理由：同格式同码率的文件，大小差异主要来自编码参数（VBR vs CBR），不代表音质差异，不值得替换。

### 6.3 Manager.EnqueueUpgrade

```go
// download/task.go

type UpgradeResult struct {
    BatchID string
    Queued  int
    Skipped int
    Errors  []UpgradeError
}

type UpgradeError struct {
    TaskID string `json:"task_id"`
    Reason string `json:"reason"`
}

// EnqueueUpgrade 将已完成的任务重新入队以升级音质。
// taskIDs 为空时升级所有已完成任务。
// quality 为目标音质（lossless/high/standard）。
func (m *Manager) EnqueueUpgrade(taskIDs []string, quality string) UpgradeResult
```

**逻辑**：

1. 如果 taskIDs 为空，遍历所有 status=done 且 skipped=false 的任务
2. 对每个任务：
   - 检查 status 是否为 done → 否则跳过，记录原因
   - 从 `m.providers` 查找 task.Source 对应的 ProviderFuncs → 找不到则跳过
   - 深拷贝 task.Song，设置 `Extra["quality"] = quality`
   - 调用 `m.Enqueue(song, source, pf.GetDownloadURL, pf.GetLyrics)`
3. 所有任务入队后，创建 batch 记录，返回 UpgradeResult

## 7. 错误处理

### 7.1 错误场景

| 场景 | 处理 | 日志级别 |
|------|------|----------|
| Provider 降级到最低仍失败 | Task 标记 failed，显示错误信息 | ERROR |
| Glob 匹配文件失败 | 当作无已有文件，正常下载 | WARN |
| 临时文件下载失败 | 删除 .tmp，返回 error，旧文件安全 | ERROR |
| Rename 失败 | 返回 error，.tmp 文件残留（下次下载会覆盖） | ERROR |
| Remove 旧文件失败 | 新文件已就位，旧文件残留，不影响功能 | WARN |
| 升级接口 task ID 无效 | 记入 errors 列表返回，不中断其他任务 | WARN |
| 升级接口 provider 不可用 | 记入 errors 列表返回 | WARN |

### 7.2 日志规范

所有音质相关日志使用 `slog`，关键字段：

```go
slog.Info("download.quality",
    "task_id", task.ID,
    "requested", "lossless",
    "actual_ext", "mp3",
    "actual_bitrate", 128,
    "degraded", true,
)

slog.Info("download.upgrade",
    "task_id", task.ID,
    "previous_ext", "mp3",
    "previous_size", 4521984,
    "new_ext", "flac",
    "action", "upgraded",
)
```

## 9. 测试策略

### 9.1 单元测试

| 函数 | 测试要点 |
|------|---------|
| `qualityScore()` | 各格式各码率的分数排序正确；flac > mp3 320k > mp3 128k > m4a |
| `WriteSongToDisk()` | 无已有文件 → ActionNew；已有低质量 → ActionUpgraded；已有高质量 → ActionSkipped |
| `Song.QualityString()` | 各组合的输出字符串正确 |
| `EnqueueUpgrade()` | 空 ID 列表 → 升级所有 done 任务；无效 ID → 记入 errors |

### 9.2 集成测试

| 场景 | 验证 |
|------|------|
| 完整下载流程 | 下载 → Task 有 actual_quality 值 → 前端显示正确 |
| 升级流程 | 先下载 mp3 → 调用升级 → 检测到 flac 可用 → 替换 → Task 显示"已升级" |
| 安全替换 | 模拟下载中断 → 旧文件仍在 → .tmp 文件被清理 |
| 全部升级 | 多首歌混合状态 → 只有 done 的被入队 → running/failed 的被跳过 |

- [ ] 下载任意歌曲，Task JSON 包含 `actual_quality` 且非空

- [ ] 历史任务卡片上有"升级"按钮，点击后新任务入队
- [ ] "全部升级"按钮入队所有已完成任务，toast 显示入队数量
- [ ] 升级替换后，旧 mp3 文件被删除，新 flac 文件存在
- [ ] 升级过程中断，旧文件完好无损
- [ ] 已有 flac 文件时重新下载，Action 为 skipped
- [ ] 浏览器标签页显示自定义 favicon
