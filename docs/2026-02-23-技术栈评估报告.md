## 三、候选方案对比

### Option A：继续 Go (Gin + Vue/React SPA)

Go 后端提供 JSON API，前端 SPA 通过 `//go:embed` 嵌入二进制，单容器部署。

- **Docker**：单静态二进制，镜像 25-40MB，`scratch` 或 `alpine` 基础镜像
- **前端**：标准 SPA 开发流程，Vite 热重载，构建产物嵌入 Go 二进制
- **NAS/文件**：`os`/`io`/`path/filepath` 原生支持，goroutine 流式下载，内存极低
- **迁移成本**：零。现有代码完全不动，在上层新增 `cmd/` `api/` `service/` `frontend/`
- **并发性能**：goroutine 天然优势，每个仅 ~2KB 栈，50-100 并发下载无压力

### Option B：重写为 TypeScript/Node.js (Next.js)

全栈 TypeScript，前后端类型共享。

- **Docker**：Node.js 镜像 150MB-1GB，`node_modules` 管理复杂
- **前端**：开发体验最佳，全栈类型共享
- **NAS/文件**：Node.js stream 背压管理复杂，大文件下载需谨慎
- **迁移成本**：极高。7,936 行全部重写，加密模块移植风险最大
- **并发性能**：单线程事件循环，I/O 可用但不如 goroutine 健壮

### Option C：重写为 Python (FastAPI)

FastAPI 后端 + 独立 SPA 前端。

- **Docker**：`python:slim` 镜像 50-200MB，需管理虚拟环境
- **前端**：同样需要独立 SPA，无 `embed` 优势
- **NAS/文件**：`pathlib` 好用，但 GIL 限制真并发，内存开销高于 Go
- **迁移成本**：极高。全部重写，动态类型增加运行时 bug 风险
- **并发性能**：GIL 限制，NAS 低内存环境 (2-4GB) 下 Python 开销是问题

### Option D：混合架构 (Go 核心 + 独立前端容器)

实质等同于 Option A，只是前端单独容器，需要 docker-compose，对 NAS 用户不友好。折叠入 Option A 讨论。

## 六、最终结论：Option A — 继续使用 Go

### 三个决定性因素

**1. 零迁移成本**

7,936 行已验证代码原封不动。加密模块 (Netease AES/RSA、Soda AES-CTR + MP4 解析) 是全项目最复杂、最脆弱的部分，任何语言重写都有引入隐蔽 bug 的高风险，且无功能收益。

**2. Navidrome 先例**

19,000+ stars 的 Navidrome 使用完全相同的架构 (Go + React，`embed` 嵌入前端，单 Docker 容器)，证明该方案在 NAS 自托管场景中成熟可行。

**3. 单二进制部署是 NAS 杀手特性**

Go `embed` 将前端编译入二进制 → Docker 镜像 25-40MB → Synology/QNAP 一键部署。NAS 用户强烈偏好单容器应用，这是其他方案无法匹敌的优势。

### Go 唯一短板及弥补方案

前端开发体验 (8/10 vs TypeScript 的 10/10)：
- 使用 `tygo` 从 Go struct 自动生成 TypeScript 类型定义
- 或编写 OpenAPI spec，用 `openapi-typescript` 生成类型
- 前端本身是标准 Vue/React SPA，开发体验与全栈 TS 项目无异

## 七、推荐目标架构

```
music-lib/
  cmd/
    server/
      main.go                  # 入口：启动 HTTP 服务
  api/
    router.go                  # Gin 路由定义
    handler_search.go          # GET  /api/search?q=...&source=...
    handler_playlist.go        # GET  /api/playlists/recommended
    handler_download.go        # POST /api/download (入队下载)
    handler_status.go          # GET  /api/downloads (队列状态，SSE 推送)
  service/
    downloader.go              # 下载队列管理器 (goroutine 池, channel 队列)
    registry.go                # Provider 注册表 (source name → MusicProvider)
```

### 关键技术选型

| 层 | 推荐 | 理由 |
|----|------|------|
| Go 版本 | **1.21+** | 获得 `slog` 结构化日志、`slices`/`maps` 标准包 |
| Web 框架 | **Gin** | 81K stars，中文生态最强，性能优异 |
| 前端框架 | **Vue 3 + Vite** | 中文社区最活跃，配合 Naive UI 快速出 UI |
| 组件库 | **Naive UI** | TypeScript 编写，Vue 3 原生，中文文档完善 |
| 类型同步 | **tygo** | Go struct → TypeScript interface 自动生成 |
| 下载管理 | goroutine 池 + channel | 5-10 worker，`io.Copy` 流式写盘 |
| 存储 | Docker volume | `docker run -v /nas/music:/downloads -p 8080:8080` |

### Dockerfile 示例 (多阶段构建)

```dockerfile
# Stage 1: 构建前端
FROM node:20-alpine AS frontend
WORKDIR /app/frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run build

# Stage 2: 构建 Go 二进制
FROM golang:1.21-alpine AS backend
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
COPY --from=frontend /app/frontend/dist ./frontend/dist
RUN CGO_ENABLED=0 go build -o /music-lib ./cmd/server

# Stage 3: 最终镜像
FROM alpine:3.19
RUN apk --no-cache add ca-certificates
COPY --from=backend /music-lib /usr/local/bin/music-lib
EXPOSE 8080
VOLUME /downloads
ENTRYPOINT ["music-lib"]
```

预期镜像大小：**30-40MB**
