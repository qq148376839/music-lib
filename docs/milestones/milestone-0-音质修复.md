# Milestone 0: 音质修复与升级

> 日期：2026-04-25
> 优先级：P0（插在 M1 之前，修复当前已损坏的核心功能）
> 预计范围：3 个任务

---

## 核心问题

用户配置了无损音质，实际拿到 128kbps m4a，系统没有任何提示。后续登录 VIP 想升级已下载的文件，做不到——文件存在就跳过。

**根因**：不是代码 bug。未登录时平台只提供低音质，Provider 按设计逐级降级。但系统两个环节缺失：
1. 降级后不告知用户实际音质
2. 无法用更高音质替换已有低音质文件

---

## 任务清单

### T1: 下载结果显示实际音质

**问题**：下载完成后，任务列表只显示"完成"，用户不知道拿到的是 flac 还是 128k mp3。

**方案**：
- Provider 的 `GetDownloadURL` 返回时，将实际获得的音质信息写入 `Song.Bitrate` 和 `Song.Ext`
- QQ：已有 rate 信息（F000=flac, M800=320k mp3, M500=128k mp3, C400=m4a），命中哪个就记哪个
- 网易：已有 br 信息（999000, 320000, 128000），命中哪个就记哪个
- 酷狗：API 返回中提取 bitrate
- 前端任务列表增加一列"实际音质"，显示格式如 `FLAC` / `320kbps MP3` / `128kbps M4A`
- 如果实际音质低于用户配置，高亮提示（如橙色文字"已降级"）

**验收条件**：
- [ ] 下载任意歌曲后，任务列表显示实际获得的格式和码率
- [ ] 配置无损但实际获得 mp3 时，显示"已降级"标识

### T2: 音质升级覆盖

**问题**：`WriteSongToDisk` 检测到文件存在（size > 0）就跳过，不比较音质。用户登录 VIP 后无法升级已有文件。

**方案（不依赖 SQLite，用启发式）**：
- 下载前检查目标目录下是否已有同名歌曲（匹配 `{Artist} - {Name}.*`）
- 如果已有文件，比较逻辑：
  1. **格式优先级**：flac > mp3 > m4a。已有 flac 且请求也是 flac → 跳过
  2. **同格式比大小**：新文件明显更大（>20%）→ 替换；差不多或更小 → 跳过
- 替换时：先下载到临时文件，成功后删旧文件、移新文件（原子操作，失败不丢旧文件）
- 前端任务列表：被替换的任务显示"已升级"标识（区别于"新下载"和"已跳过"）

**关键约束**：
- 只在"新文件音质 > 旧文件音质"时替换，绝不降级
- 替换操作必须是安全的：先写临时文件 → 成功后替换，不能出现"旧的删了新的没下完"

**验收条件**：
- [ ] 未登录下载歌曲 A（得到 128k mp3）→ 登录 VIP → 重新下载歌曲 A → 系统获取 flac 并替换旧 mp3
- [ ] 已有 flac 文件 → 重新下载 → 系统跳过（不降级）
- [ ] 替换过程中断电/断网 → 旧文件仍在，不丢失

### T2.1: 历史页面音质升级入口

**问题**：T2 的升级逻辑存在但没有 UI 触发方式。用户得重新搜索每首歌再下载，500 首歌不现实。

**方案**：
- 下载历史列表每行显示实际音质（复用 T1 的信息）
- 每首歌增加"升级"按钮：重新入队下载，T2 比较逻辑自动判断是否替换
- 增加"全部升级"按钮：批量把所有历史记录重新入队，T2 自动跳过不需要升级的
- 后端新增 API：`POST /api/nas/download/upgrade` 接受 task ID 列表，重新入队
- 升级中的任务状态显示"升级中"，完成后显示"已升级 128k→FLAC"或"无需升级"

**关键约束**：
- 不做单独的"检测可升级"步骤（需要逐首调 Provider API，太重）
- 直接重新入队，让下载+比较逻辑判断
- "全部升级"走批量接口，复用现有并发控制

**验收条件**：
- [ ] 历史页面每首歌旁有"升级"按钮，点击后重新入队下载
- [ ] "全部升级"按钮一键批量入队，自动跳过已是最高音质的文件
- [ ] 升级完成后显示结果："已升级 128k mp3 → FLAC"或"无需升级（已是最高音质）"

## 技术影响

### 需要改动的文件

| 文件 | 改动 |
|------|------|
| `qq/qq.go` | `GetDownloadURL` 命中某个 rate 时，回写 `Song.Bitrate` 和 `Song.Ext` |
| `netease/netease.go` | 同上，回写实际 br |
| `kugou/kugou.go` | 从 API 响应提取 bitrate 回写 |
| `download/writer.go` | 替换 `os.Stat` 跳过逻辑为音质比较逻辑 |
| `download/task.go` | Task 增加 `ActualQuality` 字段；升级替换状态 |
| `model/song.go` | 确认 `Bitrate` 和 `Ext` 字段可被 Provider 回写 |
| `web/js/app.js` | 任务列表显示实际音质 + 降级/升级标识 + 升级按钮 + 全部升级按钮 |
| `internal/api/handler_nas.go` | 新增 `POST /api/nas/download/upgrade` 批量升级接口 |
| `web/index.html` | favicon 引用 |
| `web/favicon.svg` | 新增 |

### 不改动的

- 搜索逻辑不动
- 下载队列调度不动
- fallback 逻辑不动（质量比较在写入层，不在获取层）
