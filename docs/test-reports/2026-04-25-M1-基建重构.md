| C2 | 重启后批量下载名称丢失 | `download/task.go:186` + `internal/api/handler_nas.go:218` | `EnqueueBatch` 创建的 synthetic batch entry（`m.tasks[batchID]`）不会被持久化到 `download_tasks` 表。重启后 `LoadTasks` 只恢复 TaskRecord，synthetic entry 丢失。`ListBatches` 查不到 batch name，退化为显示 batchID。虽然 `download_batches` 表存了 name，但 `handleListBatches` 只读内存，不查 DB。 | 方案 A: `handleListBatches` 改为从 DB 聚合（JOIN download_batches + download_tasks GROUP BY batch_id）。方案 B: `LoadTasks` 时同时加载 BatchRecord 恢复 synthetic entry。推荐 A，因为 DB 聚合才是设计初衷。 |

### Major（应该修复，影响质量）

| # | 问题 | 位置 | 描述 | 修复建议 |
|---|------|------|------|----------|
| M1 | `isRetryable` 依赖 error message 字符串匹配 | `download/task.go:457-499` | 判断 HTTP 5xx/429/404/403 靠 `strings.Contains(msg, "status 50")` 等。如果 provider 返回的错误格式不同（如 `"HTTP 500"` vs `"status 500"`），重试不会触发。 | 定义 `type HTTPStatusError struct { Code int }` 让 provider 包返回结构化错误；`isRetryable` 改为 `errors.As` 类型断言。短期: 扩展匹配模式覆盖已知 provider 的实际错误格式。 |
| M2 | `login/manager.go` 和 `download/fallback.go` 仍用 `log.Printf` | `login/manager.go:144,155,159,167,176,181` + `download/fallback.go:86,98` | 设计文档实施计划步骤 14 要求"全局替换 log.Printf → slog"。这两个文件未迁移，导致日志输出混合 text/JSON 两种格式，Docker 日志采集解析困难。 | 将 `log.Printf("[login/%s] ...")` 替换为 `slog.Info/Warn/Error("login.xxx", "platform", platform, ...)`。`fallback.go` 同理。 |
