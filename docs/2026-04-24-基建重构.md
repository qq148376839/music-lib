## 1. 概述

### 1.1 目标

让 music-lib 能在 NAS 上以 Docker 容器方式稳定运行：重启后下载历史不丢失，路由结构支撑后续 M2/M3 功能扩展，Docker 单命令部署。

### 1.2 范围

**覆盖：**
- `net/http` → Gin 框架迁移，拆分单体 `main.go`
- `download.Manager` 内存存储 → SQLite 持久化（GORM）
- Docker 多阶段构建 + docker-compose
- `log.Printf` → `slog` 结构化日志

**不覆盖：**
- 任何新功能（刮削、监控、UI 升级）
- 现有 API 的行为变更（URL、请求格式、响应格式全部不变）
- 数据库索引优化、分页
- API 版本控制

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| TaskRecord | GORM 持久化模型，存储在 `download_tasks` 表，字段为 Task 的平铺形式 |
| Task | `download.Task` 内存结构，运行时使用，含完整 Song 信息 |
| onTaskUpdate | Manager 的回调函数，状态变更时触发持久化 |
| DATA_DIR | 存放 SQLite 数据库文件的目录，对应 NAS 挂载点 |

## 2. 假设审计

### 2.1 真实约束（不可改变）

| 约束 | 依据 |
|------|------|
| CGO_ENABLED=0 | Docker 多阶段构建用 scratch/alpine 基础镜像，CGO 导致动态链接问题 |
| Go 1.21 | go.mod 已定，使用 slog 标准库的最低版本要求 |
| 现有 API URL 全部不变 | 前端 web/js/app.js 已硬编码这些路径，迁移阶段不改前端 |
| NAS 低权限环境 | 不能假设有 root，数据库文件用 0644 权限，目录 0755 |

### 2.2 行业惯例（需要审视）

| 惯例 | 审视结果 | 是否采纳 | 理由 |
|------|----------|----------|------|
| GORM 用官方 `gorm.io/driver/sqlite` | 该 driver 依赖 CGO，与约束冲突 | **否** | 改用 `github.com/glebarez/sqlite`（纯 Go GORM driver，内部用 modernc.org/sqlite） |
| 用 PostgreSQL 作持久化 | PostgreSQL 需要第二个容器；NAS 重启有启动顺序依赖；空载占 50-100MB 内存；备份从"复制文件"变成"pg_dump"；用户认证、连接池配置全要管 | **否** | 单用户、低写并发、无网络访问需求、NAS 本地部署——SQLite 的所有优势完全匹配，PostgreSQL 的优势（高并发写、网络访问、复制）在这个场景全部用不上 |
| 用 UUID 作主键 | UUID 需引入额外依赖（`github.com/google/uuid`） | **否** | 用 `crypto/rand` 生成 16 字节 hex ID，stdlib 足够 |
| `download_batches` 表作为 M3 再加 | M3 的 monitor 需要 batch 级别统计，现在建好省得改表 | **否（现在建）** | 用户明确要求现在建；M3 的 monitor_runs 表也会引用 batch，提前规范化降低后期迁移成本 |
| Handler 定义在各自业务包内 | `download/handler.go` 中的 HTTP handler 依赖 `net/http`，迁移到 Gin 后需改接口 | **否** | HTTP handler 全部迁移到 `internal/api/`，`download/` 包保持纯业务逻辑 |
| go:embed 嵌入前端 | 当前前端是 vanilla HTML/JS，embed 不需要构建步骤 | **是** | M1 直接 embed web/ 目录，Dockerfile 不需要 node 构建阶段 |

### 2.3 关键决策

| 决策点 | 选择 | 备选方案 | 为什么选 A 不选 B |
|--------|------|----------|-------------------|
| SQLite driver | `github.com/glebarez/sqlite` | `gorm.io/driver/sqlite`（CGO）、`mattn/go-sqlite3`（CGO） | 唯一同时满足纯 Go + GORM 兼容的 driver |
| 关系型存储方案 | SQLite | PostgreSQL | 见 2.2 行业惯例审视 |
| Manager 持久化方式 | 回调函数 `onTaskUpdate func(*Task)` | Manager 直接持有 `*gorm.DB` | 保持 `download/` 包无 GORM 依赖，可独立测试；startup 通过 `LoadTasks` 恢复状态 |
| Task ID | `crypto/rand` 生成 16 字节 hex，格式 `t-{16hex}` | UUID、自增 int | 不引入新依赖，重启不冲突，格式与现有前端兼容（仍是字符串） |
| 重试策略 | 主源最多 3 次（含首次）指数退避，全部失败后换源 | 只换源不重试、只重试不换源 | 网络抖动用重试解决，平台限流/版权用换源解决，两者互补；现有 `fallback.go` 已实现换源逻辑，在其前面加重试层即可 |
| 重启后 pending/running 任务处理 | 标记为 `failed`，不自动重入队 | 自动重试 | 重启时下载必然中断产生半损坏文件，重入队会覆盖部分写入；用户看到 failed 可手动重触发，比静默重试更可控 |
| 前端文件服务 | `go:embed web/*` + `router.StaticFS` | Docker COPY 到容器目录再运行时读取 | embed 让最终镜像只有单二进制，无需关心文件路径 |
| CORS 策略 | `CORS_ORIGINS` 环境变量，默认 `*` | 硬编码 | NAS 用户 IP 不固定，宽松策略更实用 |

### 2.4 翻转条件

本设计需重新评估，如果：
- NAS 环境支持 CGO（如 Synology DSM 提供 GCC），可换回 `gorm.io/driver/sqlite`（更成熟）
- 下载任务量超过 100,000 条，需要考虑分页 + 索引
- 需要多用户/多实例部署（家庭共享场景），SQLite WAL 模式的写并发限制才会成为瓶颈，那时再考虑 PostgreSQL

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────┐
│                   cmd/server/main.go                │
│  wires: store.Init() → api.NewRouter() → srv.Run()  │
└──────────────┬──────────────────────────────────────┘
               │
    ┌──────────▼──────────┐
    │   internal/api/     │    HTTP 层（Gin）
    │   router.go         │
    │   middleware.go      │   Recovery / slog / CORS
    │   handler_search.go  │
    │   handler_playlist.go│
    │   handler_login.go   │
    │   handler_nas.go     │
    │   embed.go           │   go:embed web/*
    └─────┬──────────┬────┘
          │          │
  ┌───────▼───┐  ┌───▼──────────┐
  │ download/ │  │internal/store│
  │ manager.go│  │ db.go        │  GORM + glebarez/sqlite
  │ task.go   │  │ task.go      │
  │ writer.go │  └──────────────┘
  │ fallback.go│        ↑
  └─────┬─────┘    onTaskUpdate callback
        │
  ┌─────▼──────┐
  │   model/   │  Song, Playlist
  │  provider/ │  11 个平台包（不动）
  └────────────┘
```

### 3.2 模块划分

| 模块 | 职责 | 依赖 | 包路径 |
|------|------|------|--------|
| cmd/server | 启动入口，依赖注入，wiring | internal/api, internal/store, download | `cmd/server/main.go` |
| internal/api | Gin 路由 + handler，HTTP 请求/响应处理 | download, login, model, internal/store | `internal/api/*.go` |
| internal/store | SQLite 初始化，TaskRecord CRUD | gorm, glebarez/sqlite | `internal/store/*.go` |
| download | 下载业务逻辑：队列、写盘、降级换源 | model, utils | `download/*.go`（删除 handler.go） |
| model | 共享领域模型 Song, Playlist | utils | `model/song.go` |
| 各平台 provider | 搜索/下载/歌词/歌单（不动） | model, utils | `netease/`, `qq/`, ... |
| login | QR 码登录管理（不动） | — | `login/manager.go` |

### 3.3 核心流程：下载任务的生命周期

```
前端 POST /api/nas/download
        ↓
handler_nas.go 解析请求, 调用 manager.Enqueue()
        ↓
Manager 生成 task ID (crypto/rand), 写入内存 map
onTaskUpdate(task, "") → store.Save() → upsert download_tasks
        ↓
goroutine 执行下载 (runTask)
  │
  ├─ status = running → onTaskUpdate → DB
  │
  ├─ [重试循环] withRetry(maxRetries=3, backoff=指数退避)
  │     尝试 1：getURL() → downloadFile()
  │     失败(网络/超时) → sleep(1s) → 尝试 2
  │     失败 → sleep(2s) → 尝试 3
  │     失败 → 进入换源逻辑
  │         tryFallback() 按 fallbackOrder 逐平台搜索匹配 → getURL → downloadFile
  │         换源成功: task.FallbackSource = newSource
  │         换源全败: failTask()
  │
  ├─ 下载完成: status = done → onTaskUpdate → DB
  └─ 下载失败: status = failed, error="..." → onTaskUpdate → DB

重试触发条件（触发重试）：
  - net.Error（网络超时、连接重置）
  - HTTP 5xx（平台服务器错误）
  - HTTP 429（限流，退避后重试）

不触发重试（直接换源）：
  - HTTP 404（资源不存在）
  - URL 返回为空字符串（平台拒绝/版权）
  - HTTP 403（权限问题，换源更有效）

服务重启时:
  store.MarkRunningAsFailed(db)           // running/pending → failed
  existingTasks = store.ListAll(db)
  manager.LoadTasks(existingTasks)         // 恢复历史记录到内存 map
```

### 3.4 重试配置

通过环境变量控制，有合理默认值：

| 环境变量 | 默认值 | 说明 |
|---------|--------|------|
| `DOWNLOAD_MAX_RETRIES` | `3` | 主源最大尝试次数（含首次），超过后换源 |
| `DOWNLOAD_RETRY_BACKOFF` | `2` | 指数退避基数（秒），第 n 次失败后等待 backoff^(n-1) 秒（1s, 2s, 4s） |
| `DOWNLOAD_CONCURRENCY` | `3` | 并发下载数 |

## 5. 数据模型

### 5.1 表结构

#### download_batches

| 字段 | 类型 | 约束 | 索引 | 说明 |
|------|------|------|------|------|
| id | TEXT | PRIMARY KEY | PK | `b-{16hex}` 格式 |
| source | TEXT | NOT NULL | | 来源平台 |
| name | TEXT | NOT NULL | | 歌单名或用户备注 |
| total | INTEGER | NOT NULL | | 入队时歌曲总数（快照） |
| created_at | DATETIME | NOT NULL | | |
| updated_at | DATETIME | NOT NULL | | |

说明：done/failed/running/pending 的实时计数从 `download_tasks` 按 batch_id 聚合，不存入 batch 表（避免写竞争，task 状态变更时不需要更新 batch 行）。

#### download_tasks

| 字段 | 类型 | 约束 | 索引 | 说明 |
|------|------|------|------|------|
| id | TEXT | PRIMARY KEY | PK | `t-{16hex}` 格式 |
| source | TEXT | NOT NULL | idx_source_song | 来源平台：netease/qq/... |
| fallback_source | TEXT | | | 换源后实际使用的平台 |
| batch_id | TEXT | FK→download_batches.id | idx_batch_id | 空字符串表示单曲 |
| song_id | TEXT | | idx_source_song | 平台内歌曲 ID（与 source 组成去重键） |
| title | TEXT | NOT NULL | | 歌曲标题 |
| artist | TEXT | | | 艺术家 |
| album | TEXT | | | 专辑 |
| ext | TEXT | | | 文件扩展名：mp3/flac/... |
| quality | TEXT | | | lossless/high/standard |
| file_path | TEXT | | | NAS 上的绝对路径 |
| status | TEXT | NOT NULL | idx_status | pending/running/done/failed |
| error | TEXT | | | 失败原因（含重试次数信息） |
| retry_count | INTEGER | DEFAULT 0 | | 已重试次数（不含首次尝试） |
| skipped | INTEGER | DEFAULT 0 | | 1=文件已存在跳过 |
| progress | INTEGER | DEFAULT 0 | | 已下载字节数 |
| total_size | INTEGER | DEFAULT 0 | | 文件总字节数 |
| created_at | DATETIME | NOT NULL | | 任务创建时间 |
| updated_at | DATETIME | NOT NULL | | 最后更新时间 |
| completed_at | DATETIME | | | 完成（成功或失败）时间 |

索引说明：
- `idx_status`：ListTasks 按状态过滤
- `idx_batch_id`：聚合 batch 统计
- `idx_source_song`：复合索引 `(source, song_id)`，M3 去重查询

**M3 去重查询预留：**
```sql
SELECT 1 FROM download_tasks
WHERE source = ? AND song_id = ? AND status = 'done'
LIMIT 1
```

### 5.2 GORM 模型定义

```go
// internal/store/batch.go
package store

type BatchRecord struct {
    ID        string    `gorm:"primaryKey"`

    Name      string    `gorm:"not null"`
    Total     int       `gorm:"not null"`
    CreatedAt time.Time `gorm:"not null"`
    UpdatedAt time.Time `gorm:"not null"`

func (BatchRecord) TableName() string { return "download_batches" }
```

```go
// internal/store/task.go
package store

type TaskRecord struct {
    ID             string     `gorm:"primaryKey"`
    Source         string     `gorm:"not null;index:idx_source_song,composite:source"`
    FallbackSource string
    BatchID        string     `gorm:"index"`

    Album          string
    Ext            string
    Quality        string
    FilePath       string
    Status         string     `gorm:"not null;index"`
    Error          string
    RetryCount     int        `gorm:"default:0"`
    Skipped        bool
    Progress       int64
    TotalSize      int64
    CreatedAt      time.Time  `gorm:"not null"`
    UpdatedAt      time.Time  `gorm:"not null"`
    CompletedAt    *time.Time
}

func (TaskRecord) TableName() string { return "download_tasks" }
```


### 5.4 Task ↔ TaskRecord 转换规则

`download.Task` → `store.TaskRecord`（保存时）：
- `task.Song.ID` → `record.SongID`
- `task.Song.Name` → `record.Title`
- `task.Song.Artist` → `record.Artist`
- `task.Song.Album` → `record.Album`
- `task.Song.Ext` → `record.Ext`
- `task.Song.Extra["quality"]` → `record.Quality`
- `task.RetryCount` → `record.RetryCount`
- 其余字段直接映射

`store.TaskRecord` → `download.Task`（读取时）：
- 重建 `model.Song{ID: SongID, Name: Title, Artist: Artist, Album: Album, Ext: Ext}`
- `Extra["quality"] = record.Quality`
- 注意：URL、Cover 等运行时字段不持久化，重建后为空（历史记录只用于展示）

### 5.5 关系图

```
download_batches          download_tasks
├── id (PK)     ←──────── batch_id (FK，可为空)
├── source                ├── id (PK)
├── name                  ├── source + song_id  (复合去重键)
└── total                 ├── status
                          ├── retry_count
                          └── ...

M3 新增（设计预留，M3 实现）：
monitor_runs
├── id (PK)
├── monitor_id
└── batch_id ──────────→ download_batches.id
```

外键约束：SQLite 默认关闭外键（PRAGMA foreign_keys=OFF），GORM 不自动启用。batch_id 是逻辑外键，通过应用层保证一致性（创建 task 前先创建 batch）。

## 6. 错误处理

### 6.1 错误码体系

| 范围 | 场景 | HTTP Status | code | message 示例 |
|------|------|-------------|------|-------------|
| 请求参数 | source 参数缺失/无效 | 400 | -1 | `"unknown or missing source: \"foo\""` |
| 请求参数 | keyword/id/link 缺失 | 400 | -1 | `"missing keyword parameter"` |
| 请求参数 | JSON 解析失败 | 400 | -1 | `"invalid request body: ..."` |
| 业务逻辑 | provider 不支持该功能 | 501 | -1 | `"lyrics not supported for bilibili"` |
| 业务逻辑 | NAS 未配置 | 503 | -1 | `"NAS download not configured (MUSIC_DIR not set)"` |
| 下游错误 | 平台 API 调用失败 | 500 | -1 | `"search failed: ..."` |
| 资源不存在 | task_id 不存在 | 404 | -1 | `"task not found"` |

### 6.2 下载任务错误信息格式

task.Error 字段需包含足够上下文供排查：

```
// 主源重试耗尽，换源成功
"primary source netease failed after 3 attempts (last: http status 503); fallback to kugou succeeded"

// 主源重试耗尽，换源全失败
"primary source netease failed after 3 attempts (last: network timeout); all 10 fallback providers exhausted"

// 文件写入失败（不重试，直接失败）
"write to disk: no space left on device"

// 服务重启打断
"interrupted by server restart (was running)"
```

### 6.3 Gin Panic Recovery

`middleware.Recovery()` 捕获所有 panic：
```go
// 记录 slog.Error with stack trace
// 返回 500 { "code": -1, "message": "internal server error" }
// 不暴露 panic 详情到客户端
```

### 6.4 日志规范

```go
// slog 格式（JSON，适合 Docker/NAS 日志采集）

// 服务启动
slog.Info("server starting", "port", port, "music_dir", musicDir, "data_dir", dataDir)

// 请求日志（middleware 自动记录）
slog.Info("request", "method", c.Request.Method, "path", c.Request.URL.Path,
    "status", c.Writer.Status(), "latency_ms", latency.Milliseconds())

// 下载事件
slog.Info("download.enqueue", "task_id", id, "title", song.Name, "artist", song.Artist, "source", source)
slog.Info("download.done", "task_id", id, "file", filePath, "skipped", skipped)
slog.Error("download.failed", "task_id", id, "song", song.Display(), "error", err)

// 重试
slog.Warn("download.retry", "task_id", id, "attempt", n, "max", maxRetries, "error", lastErr, "wait_ms", backoffMs)

// 换源
slog.Warn("download.fallback", "task_id", id, "from", originalSource, "to", fallbackSource, "retry_exhausted", true)

// 登录事件
slog.Info("login.success", "platform", platform, "nickname", nickname)
slog.Warn("login.error", "platform", platform, "error", errMsg)
```

日志级别控制（`LOG_LEVEL` 环境变量）：
- `debug`：全量，含 HTTP 请求详情
- `info`（默认）：业务事件
- `warn`：可恢复的异常（降级、跳过）
- `error`：需要关注的错误

**cmd/server/main.go wiring 逻辑（伪代码描述）：**
```
1. 读取环境变量（PORT, MUSIC_DIR, DATA_DIR, DOWNLOAD_CONCURRENCY, DOWNLOAD_MAX_RETRIES, 等）
2. db = store.Init(dataDir)
3. store.MarkRunningAsFailed(db)
4. existingTasks = store.ListAllTasks(db)
5. cfg = download.Config{MusicDir, Concurrency, MaxRetries, RetryBackoff}
6. dlMgr = download.NewManager(cfg, providers)
7. dlMgr.SetOnTaskUpdate(func(t *download.Task) {
       store.SaveTask(db, t)
   })
8. dlMgr.LoadTasks(existingTasks)
9. router = api.NewRouter(providers, loginMgr, dlMgr, db)
10. router.Run(":" + port)
```

注意步骤 9：`db` 也传给 router，`handler_nas.go` 的 `ListBatches` 需要从 DB 聚合 batch 统计。
