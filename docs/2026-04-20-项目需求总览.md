## 1. 项目背景

基于现有 music-lib 项目进行重写，整合三个参考项目的优点，打造一个面向 NAS 环境的完整音乐下载与管理系统。

### 1.1 现有项目
- **music-lib**（Go）：多平台音乐聚合库，11 个平台 provider，Web UI + REST API + Docker 部署。零外部依赖，代码约 8,800 行，健康分 6.5/10。

### 1.2 参考项目
| 项目 | 语言 | 代码量 | 主要借鉴点 |
|------|------|--------|-----------|
| simple_sq_music_plus | Java 17 + Spring Boot | ~54,000 行 | 插件架构（Template Method）、DB 热加载配置（100+ key）、ID3 标签管理、Emby/Jellyfin 集成、队列式下载系统 |
| go-music-dl | Go 1.25 | ~7,400 行 | SQLite + GORM 统一存储、并发多源搜索（WaitGroup）、相似度匹配换源、FFmpeg 元数据嵌入、本地收藏管理、go:embed 模板 |
| music-tag-web | Python + Django | ~5,000 行 | 多源刮削 + 智能打分（QQ/网易/咪咕/酷狗/酷我）、Acoustid 音频指纹识别、14 种音频格式标签支持、Celery 异步任务 |

### 1.3 技术决策
- **后端语言**：Go（统一架构，复用现有 provider 代码）
- **前端**：独立 Web UI，通过 REST API 与后端通信，使用 frontend-design skill 设计
- **存储**：SQLite（参考 go-music-dl）
- **部署**：Docker（面向 NAS 环境）
- **外部依赖**：放弃零依赖原则，按需引入库（GORM、Gin 等）

## 2. 核心功能需求

### 2.1 多平台搜索 + 下载
- 支持 11+ 音乐平台搜索
- 支持无损下载，如果无损不可用则自动降级
- 支持直接下载到本地（浏览器下载）
- 支持下载到 NAS（队列式后台下载）

### 2.2 歌单下载
- 解析各平台歌单链接，批量下载歌单内所有歌曲
- 支持歌单导入和管理

### 2.3 监控自动化
- 监控各大平台的推荐榜、排行榜信息
- 所有平台、所有榜单都需要支持
- 定时自动抓取并下载新歌到 NAS
- 刷新间隔可配置（6 小时 / 12 小时等）
- 下载规则可配置（Top N、去重规则等）
- 支持自定义监控（如关注特定歌手、特定歌单的更新）

### 2.4 音乐刮削
- 下载完成后自动进行元数据刮削
- **标签编辑**：歌曲标题、艺术家、专辑等 ID3 标签写入
- **歌词下载**：多平台歌词获取，LRC 格式
- **封面下载**：专辑封面获取并嵌入音频文件
- Go 原生实现（参考 music-tag-web 的多源搜索 + 智能打分逻辑）

### 2.5 平台登录
- 支持 Cookie 管理，用于下载会员/VIP 歌曲
- 重点支持：QQ 音乐、网易云音乐
- 支持 QR 码扫描登录流程

### 2.6 日志系统
- 结构化日志，详细的报错信息
- 下载任务状态追踪（成功/失败/重试）
- 错误可追溯，便于调试

### 2.7 Docker 部署
- 面向 NAS 环境（Synology 等）
- docker-compose 一键部署
- 数据持久化（音乐文件、配置、数据库）
- 资源占用合理

## 3. 角色分工

| 角色 | 指令 | 职责 | AI 参与度 |
|------|------|------|----------|
| 架构师 | /design | 需求 → 设计文档（docs/YYYY-MM-DD-中文名.md） | AI 出完整设计 → 用户审批 |
| 工程师 | /dev | 设计文档 → Go 代码 | AI 全权写码，设计模糊处暂停请示 |
| 前端设计师 | /ui | API 规格 → Web UI（frontend-design skill） | AI 全权设计 + 实现 |
| 质检官 | /test /review /deploy | 测试 + Code Review + Docker 部署验证 | AI 全权测试 + 审查 |

### 工作流
```
用户提需求
    ↓
/design → 架构师（输出设计文档）
    ↓
用户确认设计
    ↓
/dev → 工程师（写 Go 代码）  ┐
/ui  → 前端设计师（做 Web UI）┘ 可并行
    ↓
/test → 质检官（跑测试 + 审查）
    ↓
/deploy → 质检官（Docker 部署验证）
    ↓
用户验收
```

## 5. 贯穿原则

**第一性原理审计**贯穿所有角色的所有工作：
- 架构师：设计文档包含「假设审计」章节，区分真实约束 vs 行业惯例
- 工程师：实现前审计设计假设，过时假设打回架构师
- 前端设计师：UI 方案审计用户真正需要的操作，不照搬
- 质检官：Review 时检查未审计的假设，测试覆盖「惯例不成立」的场景
