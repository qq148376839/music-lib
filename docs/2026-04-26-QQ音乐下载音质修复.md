## 1. 概述

### 1.1 目标

修复 QQ 音乐会员用户无法下载高音质（FLAC/320k）的问题——根因是 MQTT QR 登录产出的 `qqmusic_key` 凭证与 GetVkey API 请求参数不匹配，导致服务端将请求视为匿名。

### 1.2 范围

| 覆盖 | 不覆盖 |
|------|--------|
| `qq/qq.go` GetDownloadURL 请求参数修复 | 前端 UI 变更 |
| 新增 zzb 签名算法（Go 实现） | QR 登录流程改动 |
| 自动检测认证类型（qqmusic_key vs OAuth） | 网易云/酷狗等其他平台 |
| GUID 生成格式修正 | 数据库 schema 变更 |

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| `qqmusic_key` | MQTT QR 登录产出的应用端认证令牌 |
| `authst` | QQ 音乐应用端 API 中传递认证令牌的参数名 |
| `g_tk` | QQ 音乐网页端 CSRF token，由 `skey`/`p_skey` 计算 |
| `ct` | client type，11=应用端，24=网页端 |
| `zzb` sign | QQ 音乐 API 请求签名算法（MD5 + 字节提取 + XOR + Base64） |
| GetVkey | `music.vkey.GetVkey.UrlGetVkey`，获取歌曲下载地址的核心 API |

## 2. 假设审计

### 2.1 真实约束（不可改变）

| 约束 | 依据 |
|------|------|
| MQTT QR 登录产出 `qqmusic_key`，不含 `p_skey` | qr_login.go 第 302-308 行实现；Login 交换返回 code=1000 已确认失败 |
| QQ 音乐 GetVkey API 对 `ct:24`（网页端）+ `qqmusic_key` 认证不返回高音质 purl | 用户实测：会员账号只能下到最低音质 |
| `simple_sq_music_plus` 使用 `ct:"11"` + `authst` 能正常获取 FLAC | 参考项目 Java 实现（QQMusicUtil.java, QQHander.java）已验证可用 |

### 2.2 行业惯例（需要审视）

| 惯例 | 审视结果 | 是否采纳 | 理由 |
|------|----------|----------|------|
| 请求需要 sign 签名 | musicu.fcg 支持无签名请求，但带签名更稳定 | 采纳 | 52pojie 分析确认 musicu.fcg 可不签名，但参考实现全部带签名；签名成本低（纯 MD5 运算），加上更安全 |
| 使用 `musics.fcg`（纯加密端点） | 不采纳 | 不采纳 | musics.fcg 需要额外的 ag-1 加解密，复杂度高且收益不明确 |
| 同时维护网页端和应用端两套参数 | 采纳 | 采纳 | 未来可能接入传统 OAuth（p_skey）登录，保留兼容性 |

### 2.3 关键决策

| 决策点 | 选择 | 备选方案 | 为什么选 A 不选 B |
|--------|------|----------|-------------------|
| 认证模式检测 | 从 cookie 内容自动检测 | 用户手动选择 | cookie 中有 `qqmusic_key` 就用应用端模式，有 `p_skey` 就用网页端模式；零配置，不增加用户负担 |
| sign 算法实现 | Go 原生实现 zzb 签名 | 调用外部命令/服务 | 算法简单（MD5 + 固定字节操作），无需外部依赖 |
| 请求体 key 命名 | 应用端用 `"music.vkey.GetVkey.UrlGetVkey"`，网页端保留 `"req_1"` | 统一用一种 | 参考实现验证应用端必须用全路径 key |
| User-Agent | 应用端用 QQ 音乐客户端 UA | 统一用通用 UA | 服务端可能根据 UA 判断客户端类型 |

### 2.4 翻转条件

- 假设 1：QQ 音乐不封杀 zzb 签名算法。如果 QQ 强制升级到 zzc/ag-1，本方案需要更新签名逻辑。
- 假设 2：`musicu.fcg` 端点持续可用。如果 QQ 全面迁移到 `musics.fcg`，需要实现 ag-1 加解密。
- 假设 3：MQTT QR 登录产出的 `qqmusic_key` 具有完整的 VIP 下载权限。如果该 token 权限受限，需要修复 Login 交换流程（code=1000 问题）。

## 3. 架构设计

### 3.1 整体架构

改动范围集中在 `qq` 包内部，不影响 API 层、下载管理器、前端。

```
前端 → API handler → download.Manager → qq.GetDownloadURL()
                                              ↓
                                    [改动区域] 构建请求参数
                                              ↓
                               musicu.fcg?sign=xxx (POST)
                                              ↓
                                     解析 purl → 返回下载 URL
```

### 3.2 模块划分

| 模块 | 职责 | 依赖 | 包路径 |
|------|------|------|--------|
| 认证模式检测 | 从 cookie 判断使用应用端还是网页端参数 | 无 | `qq/qq.go` 内部方法 |
| zzb 签名 | 对请求 JSON 计算 sign | `crypto/md5`, `encoding/base64` | `qq/sign.go`（新文件） |
| GetVkey 请求构建 | 根据认证模式组装不同的请求参数 | 认证检测、zzb 签名 | `qq/qq.go` GetDownloadURL |
| GUID 生成 | 生成 32 位十六进制随机字符串 | `math/rand` | `qq/qq.go` 内部方法 |

### 3.3 核心流程

**改造后的 GetDownloadURL 流程：**

```
1. 检测 cookie 认证类型
   ├─ 含 qqmusic_key → appAuth 模式
   └─ 含 p_skey/skey → webAuth 模式（现有逻辑保留）

2. [appAuth] 构建请求体
   ├─ comm: ct="11", cv=13020508, authst=<key>, qq=<uin>, tmeLoginType=<type>
   ├─ key: "music.vkey.GetVkey.UrlGetVkey"
   └─ guid: 32 位 hex

3. [webAuth] 构建请求体（现有逻辑）
   ├─ comm: ct=24, cv=4747474, g_tk=<computed>, uin=<uin>
   ├─ key: "req_1"
   └─ guid: 随机数字

4. 计算 zzb sign（两种模式都加）

5. POST musicu.fcg?sign=<sign>&signature=<sign>
   ├─ Cookie header: 原始 cookie
   ├─ User-Agent: 根据模式选择
   └─ Body: JSON 请求体

6. 解析响应
   ├─ appAuth 模式响应 key: "music.vkey.GetVkey.UrlGetVkey"
   └─ webAuth 模式响应 key: "req_1"

7. 音质降级逻辑（保持不变）
```

## 4. API 规格

本次修复不涉及对外 API 变更。所有改动在 `qq` 包内部。

### 4.1 QQ 音乐 GetVkey API 请求规格（应用端模式）

**Endpoint**: `POST https://u.y.qq.com/cgi-bin/musicu.fcg?sign={sign}&signature={sign}`

**Headers**:
```
Content-Type: application/json
Referer: https://y.qq.com
User-Agent: QQ%E9%9F%B3%E4%B9%90/73222 CFNetwork/1406.0.3 Darwin/22.4.0
Cookie: uin={uin}; qqmusic_key={key}; qm_keyst={key}
```

**Request Body**:
```json
{
  "comm": {
    "cv": 13020508,
    "v": 13020508,
    "ct": "11",
    "tmeAppID": "qqmusic",
    "format": "json",
    "inCharset": "utf-8",
    "outCharset": "utf-8",
    "qq": "{uin}",
    "authst": "{qqmusic_key}",
    "tmeLoginType": "{loginType}"
  },
  "music.vkey.GetVkey.UrlGetVkey": {
    "module": "music.vkey.GetVkey",
    "method": "UrlGetVkey",
    "param": {
      "filename": ["{prefix}{songmid}{songmid}.{ext}"],
      "guid": "{32位hex}",
      "songmid": ["{songmid}"],
      "songtype": [0],
      "uin": "{uin}",
      "loginflag": 1,
      "platform": "20"
    }
  }
}
```

**Response** (成功):
```json
{
  "music.vkey.GetVkey.UrlGetVkey": {
    "code": 0,
    "data": {
      "midurlinfo": [
        {
          "purl": "C400xxxxxxx.m4a?vkey=...&guid=...&uin=...",
          "wifiurl": "http://ws.stream.qqmusic.qq.com/C400xxx...",
          "vkey": "...",
          "ekey": "...",
          "result": 0
        }
      ],
      "sip": ["http://ws.stream.qqmusic.qq.com/"]
    }
  }
}
```

**Response** (失败/无权限):
```json
{
  "music.vkey.GetVkey.UrlGetVkey": {
    "code": 0,
    "data": {
      "midurlinfo": [
        {
          "purl": "",
          "result": 301
        }
      ]
    }
  }
}
```

## 5. 数据模型

不涉及数据库变更。

### 5.1 内部数据结构变更

#### authMode 类型

`qq/qq.go` 中新增内部结构，不对外暴露：

| 字段 | 类型 | 说明 |
|------|------|------|
| mode | string | `"app"` 或 `"web"` |
| uin | string | 用户 QQ 号 |
| authst | string | qqmusic_key 值（app 模式） |
| gtk | int | g_tk 值（web 模式） |
| loginType | string | 登录类型，默认 `"2"` |

#### Cookie 持久化变更

`qq/login.go` 中 cookieFile 结构需扩展：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| Cookie | string | 必填 | cookie 原始字符串 |
| Nickname | string | 可空 | 用户昵称 |
| LoginType | string | 可空，默认 `"2"` | 登录类型（新增） |

> 为什么扩展 LoginType：MQTT 登录时 cookies 中包含 `tmeLoginType`，需要持久化以便重启后使用。qr_login.go 已能获取此值。

## 6. 错误处理

### 6.1 错误场景

| 场景 | 当前行为 | 改造后行为 |
|------|----------|-----------|
| cookie 为空 | uin="0", gtk=5381, 请求匿名 | 不变（继续降级到免费音质） |
| qqmusic_key 过期 | 高音质 purl 为空，降级 | 不变（降级），但日志中增加 `auth_mode=app, purl_empty=true` 便于诊断 |
| sign 计算失败 | N/A（无签名） | 降级到不带 sign 的请求（log warn） |
| 响应 JSON key 不匹配 | 解析失败 | 根据 auth mode 自动选择正确的响应 key |

### 6.2 日志规范

| 日志点 | 级别 | 关键字段 |
|--------|------|----------|
| 认证模式检测结果 | Info | `qq.auth_mode`, `mode`, `uin` |
| 单次音质尝试结果 | Debug | `qq.vkey_attempt`, `rate`, `purl_empty`, `result_code` |
| 音质降级发生 | Warn | `qq.quality_fallback`, `from`, `to`, `song_mid` |
| 签名计算 | Debug | `qq.sign`, `sign_len` |

### 8.3 zzb 签名算法规格

算法输入：请求 JSON 字符串
算法输出：签名字符串（如 `zzb...`）

步骤：
1. 对 JSON 做 MD5 → 得到 32 字节大写十六进制字符串
2. `head`：取索引 [21,4,9,26,16,20,27,30] 处的字节
3. `tail`：取索引 [18,11,3,2,1,7,6,25] 处的字节
4. `middle`：每两个十六进制字符解析为一个字节（高位*16+低位），与固定数组 `[212,45,80,68,195,163,163,203,157,220,254,91,204,79,104,6]` 逐字节 XOR
5. 拼接：`"zzb"` + head 各字节转 char + Base64(middle) + tail 各字节转 char
6. 转小写，移除 `/`, `+`, `=`

固定常量：
- Head 索引：`[21, 4, 9, 26, 16, 20, 27, 30]`
- Tail 索引：`[18, 11, 3, 2, 1, 7, 6, 25]`
- Middle XOR 表：`[212, 45, 80, 68, 195, 163, 163, 203, 157, 220, 254, 91, 204, 79, 104, 6]`
